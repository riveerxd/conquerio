# Project Rules

## Git
- NEVER add Co-Authored-By tags to commits. Ever. No exceptions.
- NEVER commit or push without explicit permission from the user.

## Workflow
- Always create a GitHub issue before starting work on anything.
- Issues must have appropriate labels (backend, frontend, bug, enhancement, etc.) and be assigned to the correct person per CONTRIBUTING.md.
- Follow CONTRIBUTING.md for branching, commits, and PRs.

## Code & Writing Style - MUST Follow
- All output must read as 100% human-written. No exceptions.
- NEVER use emdashes (—). Regular dashes (-) are fine.
- NEVER write overly polished, corporate, or "AI-sounding" text in code comments, commit messages, PR descriptions, or issues.
- Keep comments casual, short, and natural - like a real dev wrote them.
- No filler phrases like "This function handles...", "This method is responsible for...", "Leverage", "Utilize", "Streamline", "Robust", "Comprehensive", "Facilitate".
- Commit messages: short, lowercase, no periods. Follow CONTRIBUTING.md examples exactly.
- PR descriptions and issue text: concise, direct, no fluff.
- Variable/function names should match the existing codebase style, not introduce new conventions.
- Don't add unnecessary comments, docstrings, or annotations to code you didn't write.
- Match the tone and style of the existing codebase exactly.
